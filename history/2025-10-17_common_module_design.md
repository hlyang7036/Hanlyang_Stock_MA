# 공통 모듈 설계 및 개발 계획

## 날짜
2025-10-17

## 작업 내용

### 1. 프로젝트 구조 분석 및 공통 모듈 필요성 확인
- **현재 상황**
  - `src/strategy/`: 실시간 전략 실행 (특정 시간에 전략 실행 → 매매 → 결과)
  - `src/backtest/`: 과거 데이터 백테스팅 (기간별 반복 실행 → 성과 분석)
  
- **공통 모듈 필요성**
  - 두 프로세스에서 동일하게 사용되는 로직 존재
  - 중복 코드 방지 및 유지보수성 향상을 위해 모듈화 필요

### 2. 공통 모듈 식별 및 개발 우선순위 결정

#### 우선순위 1: 데이터 수집 모듈 (`src/analysis/data/`)
**필요성**: 모든 분석의 기초 데이터 제공
- **data_collector.py**: 데이터 수집 인터페이스
  - `get_historical_data()`: 과거 데이터 조회 (backtest용)
  - `get_realtime_data()`: 실시간 데이터 조회 (strategy용)
  - `get_market_data()`: 전체 시장 데이터 조회
- **data_preprocessor.py**: 데이터 전처리
  - 결측치 처리
  - 데이터 표준화/정규화
  - 시계열 정렬

#### 우선순위 2: 기술적 지표 계산 모듈 (`src/analysis/technical/`)
**필요성**: 전략의 핵심 지표 계산
- **moving_average.py**: 이동평균선 계산
  - SMA (5일, 20일, 40일)
  - 골든크로스/데드크로스 판단
- **macd.py**: MACD 지표 계산
  - MACD 3종 조합 (상/중/하)
  - 히스토그램, 시그널선 계산
- **atr.py**: ATR (Average True Range) 계산
  - 변동성 측정
  - 포지션 사이징 기초 데이터

#### 우선순위 3: 스테이지 판단 모듈 (`src/analysis/stage/`)
**필요성**: 매매 타이밍 결정
- **stage_analyzer.py**: 6단계 스테이지 분석
  - 이동평균선 배열 기반 스테이지 판단
  - MACD 기반 스테이지 전환 감지
- **signal_generator.py**: 매매 신호 생성
  - 진입 신호 (통상/조기 매수/매도)
  - 청산 신호 (3단계: 히스토그램/MACD선/교차)

#### 우선순위 4: 포지션 및 리스크 관리 모듈 (`src/utils/position/`)
**필요성**: 자금 및 리스크 관리
- **position_manager.py**: 포지션 관리
  - 유닛 계산 (계좌잔고 × 1% / ATR)
  - 포지션 사이징
  - 포트폴리오 제약 관리
- **risk_manager.py**: 리스크 관리
  - 손절 규칙 적용 (추세/변동성 기반)
  - 최대 포지션 제한
  - 상관관계 기반 포지션 조정

#### 우선순위 5: 주문 실행 모듈 (`src/utils/order/`) - 선택적
**필요성**: strategy 전용 (backtest는 시뮬레이션)
- **order_executor.py**: 주문 실행 추상화
  - 실거래/모의거래 분기
  - 주문 유형별 실행 (시장가/지정가)
  - 주문 결과 추적

### 3. 모듈 간 의존성 및 인터페이스 설계

```
데이터 수집 모듈
    ↓
기술적 지표 계산 모듈
    ↓
스테이지 판단 모듈
    ↓
포지션/리스크 관리 모듈
    ↓
주문 실행 모듈 (strategy only)
```

### 4. 개발 원칙
- **단일 책임 원칙**: 각 모듈은 하나의 명확한 책임만 가짐
- **의존성 역전**: 구체적인 구현이 아닌 추상화에 의존
- **테스트 가능성**: 각 모듈 독립적으로 테스트 가능
- **확장성**: 새로운 지표나 전략 추가 시 기존 코드 수정 최소화

## 이슈
- 공통 모듈의 범위 설정: 너무 세분화하면 복잡도 증가, 너무 통합하면 유연성 감소
- 백테스트와 실거래의 시간 처리 차이: 추상화 레벨 결정 필요

## 참고 사항
- 각 모듈은 명확한 입출력 인터페이스 정의 필요
- 모든 공통 모듈은 충분한 단위 테스트 작성
- 문서화(docstring) 필수
- 성능을 고려한 데이터 처리 (특히 백테스트 시)
